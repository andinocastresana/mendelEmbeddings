# Arquitectura web de mendelEmbeddings — v0.1

> **Estado**: borrador inicial. Es la primera versión "viva" después de la sesión de diseño del 2026-05-20. Cuando se ajuste, se crea una `v0.2_<fecha>_arquitectura_web.md` al lado de este archivo. **No se edita retroactivamente** — las versiones son snapshots inmutables; los cambios viven en versiones nuevas.

## TL;DR (1 párrafo)

El proyecto se monta como **servicio web público** con arquitectura **híbrida cliente/servidor**: inference facial corre en el browser del usuario (privacidad por diseño, latencia baja, costo de server bajo), y el servidor existe solo para "refinamiento avanzado" a pedido de usuarios registrados. Se desarrollan en paralelo **dos tracks**: (1) *vitrina equipos de fútbol*, completamente estática y curada por Diego, y (2) *comparador de rostros niño-padres*, que arranca 100% en cliente y se extiende con server-side opcional. Casi todo el desarrollo se puede hacer **en local** durante muchos meses antes de necesitar un host.

## 1. Cómo llegamos acá

### 1.1 Versión preliminar (mañana del 2026-05-20)

Recomendación inicial: stack tradicional "FastAPI server + SPA frontend separada + Postgres + Redis + ARQ + auth managed + Fly.io". Plan **Fase A→B→C** secuencial: motor primero (Tarea #1, completada hoy), después app primaria niño-padres, después app secundaria equipos, después escala.

### 1.2 Ajuste de la tarde (este documento v0.1)

El usuario aportó **tres ajustes que reformaron el plan**:

1. **"Quiero correr los tests localmente hasta tener un MVP que desplegar."** → Toda la arquitectura debe ser local-first, con `docker-compose` para los pocos servicios que requieran daemon.

2. **"Empezar por la vitrina equipos, no por niño-padres. Mientras tanto, ir desarrollando ambas en paralelo."** → Las dos apps son ahora **tracks paralelos**, no fases secuenciales. La vitrina es estática y sin necesidad de auth/storage/queue → no requiere casi nada de la complejidad backend.

3. **"¿Hay posibilidad de correr modelos directamente en el cliente? Híbrida: visualización casi instantánea 100% privada en cliente + refinamiento opcional en server."** → Cambia el diseño: el server pasa de ser el corazón del producto a ser una **opción premium para usuarios registrados**. Privacidad pasa de "polícita" a "diseño".

Estos tres ajustes simplificaron 6 de las 8 decisiones abiertas (W1–W8) — la mayoría quedan diferidas a la fase de escala.

## 2. El plan en 2 tracks paralelos

### Track 1 — Vitrina equipos de fútbol

**Caso de uso**: comparar selecciones / planteles, generar matrices de similitud, dendrogramas, boxplots intra-grupo. Datos curados exclusivamente por Diego.

**Stack**:

- Cliente Vite + React (SPA).
- Datos: un único JSON pre-calculado con embeddings de cada jugador (generado offline por un script Python que usa el motor `phyloface`).
- Hosting: estático (Cloudflare Pages / Netlify / GH Pages — sin decidir).
- **No requiere**: server, DB, auth, storage, cola async.

**Run local**: `cd client && npm run dev` + cuando agrega fotos: `python3 scripts/build_teams_embeddings.py`. Listo.

**Cuándo salta a hosting**: cuando se quiere compartir el URL con alguien.

### Track 2 — Comparador de rostros niño-padres

Este track tiene **dos sub-tracks** que se construyen en orden:

#### Track 2a — Cliente "puro" (anónimo, sin server)

Inference facial **100% en el browser** vía:

- **MediaPipe Tasks for Web** (Google, JS oficial) — face detection + 468 landmarks. Equivalente directo al subpaquete `phyloface.landmarks`.
- **ONNX Runtime Web** — corre el mismo modelo `w600k_r50.onnx` que usa el motor Python, en el browser (WebGPU > WebGL > WASM como fallbacks).
- Comparación por regiones: portar `phyloface.regions` y `phyloface.comparator_regional` a JS (operaciones puras sobre arrays — son ~300 líneas, factibles).

**Resultado para el usuario anónimo**:

| Feature | En cliente? | Latencia esperada |
|---------|-------------|-------------------|
| Score global (coseno entre dos embeddings) | ✅ | ~10ms |
| Comparación visual simple por región (grayscale + coseno) | ✅ | ~100ms |
| Embedding por región Nivel B (re-aplica modelo a 12 crops) | ✅ con WebGPU | ~2-3s |
| Visualizaciones (heatmap, radar, overlay de regiones) | ✅ | ~10ms |
| Occlusion sensitivity / saliency maps | ⚠ Posible pero lento (10-60s) → mejor server | — |
| Calibración con dataset de validación | ❌ Server | — |

**Imágenes y datos personales**: **nunca salen del browser**. Privacidad por diseño.

#### Track 2b — Server-side "refinamiento" (registrado opt-in)

Cuando el usuario registrado pide "comparación avanzada":

- El cliente envía `{image, embeddings_precalculated, landmarks_precalculated}` al server.
- El server **valida** que sus embeddings coinciden con los del cliente (sanity check de integridad — detecta browsers con WebGPU buggy).
- El server **salta** detección + embedding global (ya hechos) y solo corre lo que el cliente no puede:
  - Embedding por región Nivel B (más preciso aplicando regiones canónicas calibradas).
  - Occlusion sensitivity completa (60s-2min de inferencias).
  - Calibración del score contra umbral data-driven (Tarea #6).
  - Persistencia del historial.
- Resultado: refinamiento + score calibrado + heatmap explicativo.

**Ahorro operativo concreto**: como el cliente ya hizo detección + embedding global (≈30% del trabajo total), el server reduce su carga ~30% por request → más usuarios concurrentes con menos workers.

## 4. Stack concreto recomendado

### Cliente (ambos tracks)

| Capa | Tecnología | Justificación |
|------|------------|---------------|
| Build / dev server | **Vite** | Más rápido que CRA/Next para SPA; HMR instantáneo |
| UI framework | **React 18+** | Más recursos, ecosystem maduro, mejor para reclutamiento futuro |
| Routing | **React Router** o **TanStack Router** | Track 1 (`/teams`) y Track 2 (`/compare`) viven en la misma SPA |
| State (server-side) | **TanStack Query** | Cache, retries, polling, mutations — perfecto para Track 2b |
| State (local UI) | **Zustand** o `useState/useReducer` | Simple, sin boilerplate |
| Estilos | **Tailwind CSS** | Velocidad de iteración, sin context-switch a CSS files |
| Viz | **D3.js** (heatmap, dendrograma) + **Recharts** (boxplot, radar) | D3 para lo custom, Recharts para lo standard |
| Inference (Track 2a) | **ONNX Runtime Web** + **MediaPipe Tasks for Web** | Mismo modelo que Python (`w600k_r50.onnx`), mismo backbone de landmarks |

### Pipeline offline (Track 1)

| Componente | Tecnología |
|------------|------------|
| Script de generación | Python + paquete `phyloface` existente |
| Output | JSON con `{player_id, embedding[512], metadata}` por jugador |
| Trigger | Manual (Diego lo corre cuando agrega fotos) |

### Server (solo Track 2b, sin urgencia)

| Capa | Tecnología | Cuándo |
|------|------------|--------|
| HTTP framework | **FastAPI** | Track 2b |
| Cola async | **ARQ** (Redis-backed) | Track 2b |
| ORM | **SQLAlchemy** + **Alembic** (migrations) | Track 2b |
| Auth | **Clerk** o **Supabase** (managed) | Track 2b |
| Object storage | **Cloudflare R2** o **Backblaze B2** | Track 2b |

### Infraestructura local

| Componente | Tecnología | Cuándo entra |
|------------|------------|--------------|
| Postgres + Redis | **docker-compose** | Track 2b — un solo `docker compose up -d` y listo |
| Tunnel para demos | **Cloudflare Tunnel** o **ngrok** | Cuando quiero mostrar el localhost a alguien sin deploy |

## 6. Privacidad / legal

### 6.1 Marco regulatorio

- **RGPD (UE)**: aplica si hay usuarios europeos. Las caras y los embeddings faciales son **datos biométricos** (categoría especial, art. 9) → consentimiento explícito + propósito específico + derecho al borrado + minimización.
- **Ley 25.326 (Argentina)**: similar. Caras = datos sensibles. Inscripción en el Registro Nacional de Bases de Datos si se guardan.

### 6.2 Cómo el diseño aborda esto

| Caso | Tratamiento |
|------|-------------|
| Usuario anónimo (Track 2a) | Imágenes y embeddings **nunca salen del browser**. No hay base de datos a inscribir. No hay consentimiento porque no hay procesamiento de datos en el server. |
| Usuario registrado, comparación normal (Track 2a) | Igual que anónimo. El registro existe para acceder a Track 2b, no para registrar comparaciones normales. |
| Usuario registrado, refinamiento Track 2b | Consentimiento explícito al momento de pedir refinamiento (no enterrado en T&C generales). Imagen procesada y borrada salvo opt-in separado para historial. Embeddings persisten si el usuario opta-in. Retención máx 90 días por default. Right to be forgotten = DELETE simple. |

### 6.3 Caso especial Track 1 (vitrina equipos)

Jugadores de fútbol profesional son **personas públicas** en contexto de su actividad → cierta latitud bajo "interés público" y "datos manifiestamente publicados por el titular". Pero **igual** son datos biométricos. Mitigación:

- Solo procesamos fotos públicas oficiales (rosters, prensa).
- No se almacena la cara del jugador como imagen en el servidor; solo el embedding (que en este contexto está dentro del propósito declarado: comparación entre miembros del mismo grupo público).
- Política de privacidad explícita: "datos públicos, procesados para análisis estadístico de similitud, sin almacenamiento de imágenes en server".

### 6.4 Acciones concretas pendientes

- Redactar Política de Privacidad simple (1 página).
- Redactar Términos de Servicio (1 página).
- Si se monta Track 2b en producción: inscripción en Registro Nacional de Bases de Datos (AR) — trámite simple, gratuito.

## 9. Riesgos identificados / cosas a validar

### 9.1 Riesgos técnicos

| Riesgo | Mitigación / spike |
|--------|---------------------|
| **ONNX Runtime Web no carga `w600k_r50.onnx`** o da embeddings diferentes que Python | **Spike Track 2a inicial** (`scripts/verify_onnx_web_parity.py` + test JS con la misma imagen). **Si falla, todo el plan híbrido se cae** → habría que volver al plan original con server-only inference. |
| Latencia del modelo en cliente sin WebGPU (Safari, Firefox) | Fallback a server-side para esos browsers, o aceptar 10x más lento en WASM. |
| 50MB de modelo a descargar primera vez = experiencia inicial lenta | Service Worker para cache + UI claro ("preparando modelo, primera vez tarda ~30s"). |
| RAM del browser con modelo cargado | Validar en spike — probablemente OK con WebGPU/WebGL backends. |

### 9.2 Riesgos de producto

| Riesgo | Mitigación |
|--------|------------|
| El JSON de equipos crece y se vuelve pesado al cargar | Particionar por equipo / país; lazy load por sección de la vitrina. |
| Embeddings exportados a JS divergen del motor Python con el tiempo | Test de paridad en CI cuando exista (compara embeddings JS vs Py sobre fixture). |
| RGPD: usuario quiere borrado de su historial | Track 2b debe tener endpoint DELETE explícito desde día 1. |

### 9.3 Riesgos de proceso

| Riesgo | Mitigación |
|--------|------------|
| Avanzar en paralelo en dos tracks sin coordinación → divergencia de código compartido | Mantener `client/src/lib/` como zona común con tests; nada de copy-paste entre `apps/teams/` y `apps/compare/`. |
| Esta versión v0.1 se "olvida" cuando empezamos a tirar código y se ajustan cosas | Cuando se ajuste, crear `v0.2_<fecha>_arquitectura_web.md` (no editar v0.1). Releer este doc al inicio de cada sesión web. |

## 10. Próximos pasos inmediatos

**Próxima sesión** (decidido): **Track 2a, spike de viabilidad ONNX Runtime Web**.

Plan de la sesión:

1. Crear `client/` con Vite + React + TypeScript (setup mínimo).
2. Bajar `w600k_r50.onnx` del modelo InsightFace (`buffalo_l/w600k_r50.onnx`, ~92MB).
3. Crear `scripts/verify_onnx_web_parity.py`: corre el motor Python sobre una cara de prueba, guarda el embedding como JSON reference.
4. En el cliente: cargar el ONNX con `onnxruntime-web`, pasar la misma cara (subida o de `public/`), obtener embedding, comparar con el reference JSON.
5. Criterio de éxito: `cosine_similarity(emb_py, emb_js) > 0.9999` y `max(|diff|) < 1e-3` (cuasi-bit-identical aceptable; diferencias < 1e-5 si la cuantización del modelo es FP32 puro).

**Si el spike funciona** → seguimos con Track 2a real (alineación, regiones, viz).
**Si el spike falla** → volvemos a v0.2 con plan revisado (probable: server-side inference + cliente solo viz).

## 11. Apéndice — Glosario

- **SPA** (Single Page Application): app web cargada una sola vez; routing y datos via JS, sin recargas de página.
- **SSR** (Server-Side Rendering): cada página se renderiza en el server (Next.js, Nuxt). Bueno para SEO y carga inicial; no necesario acá.
- **ONNX** (Open Neural Network Exchange): formato standard para modelos de ML. Permite correr el mismo modelo en Python, JS, C++, mobile.
- **ONNX Runtime Web (`onnxruntime-web`)**: librería JS para correr modelos ONNX en browser con backends WebGPU / WebGL / WASM.
- **WebGPU**: API moderna del browser para acceso a GPU (Chrome ✅, Safari 17+ ✅, Firefox parcial). Necesaria para inference rápida.
- **WASM** (WebAssembly): binary format que corre en browser a ~80% velocidad nativa. Fallback de ONNX Runtime Web cuando no hay GPU.
- **ARQ**: librería Python de cola de tareas async (basada en Redis), alternativa moderna y minimal a Celery.
- **Spike**: un experimento corto y acotado para validar viabilidad de algo antes de invertir en construirlo completo.
- **MediaPipe Tasks for Web**: SDK de Google para correr modelos de visión (face detection, landmarks, hand tracking) en browser.
- **TanStack Query** (ex React Query): librería de state-server para frontend; cache, retries, polling.
- **TLS / HTTPS local**: requerido por WebRTC, getUserMedia (cámara), service workers. Vite lo provee con `--https` flag.
- **Magic link**: link de un solo uso enviado por email para login sin password.
